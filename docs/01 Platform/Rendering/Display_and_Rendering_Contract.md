# Display and Rendering Contract

This document defines HW5 display ownership, Sharp Memory LCD behavior, framebuffer semantics, flush behavior, DMA-safe buffer placement, low-power display behavior, and the planned LPBAM display-animation path.

Display is Platform-owned. Engine and Reference Game code request scene/frame presentation or animation capabilities; they must not control SPI, DMA, LPBAM, level translation, EXTCOMIN, SRAM4 placement, or panel-native row payloads directly.

## Hardware Path

| Item | HW5 Selection |
|---|---|
| Display panel | Sharp `LS013B7DH05` Memory LCD |
| Native panel orientation | portrait |
| Native resolution | `144 x 168` |
| Logical project orientation | landscape |
| Logical resolution | `168 x 144` |
| Pixel format | 1-bit monochrome |
| Bus | `SPI3` TX-only master |
| DMA | `LPDMA1_CH0`, `LPDMA1_REQUEST_SPI3_TX` |
| Level translator | `TXU0104RUTR` |
| Translator OE | `PD2` `VLT_LCD` |
| EXTCOMIN / VCOM | `PC13` `LCD_1HZ`, RTC 1 Hz calibration output |

## Ownership

- `thDisplay` is the sole owner of SPI3, display LPDMA, NSS behavior, framebuffer flushes, dirty tracking, and `VLT_LCD`.
- `thPower` coordinates RTC and low-power policy needed for `LCD_1HZ`.
- Other threads submit display requests only through `qDisplayCmd`.
- No direct display HAL/LL calls are allowed outside `thDisplay`.
- Engine/Reference Game code renders into approved abstractions and requests presentation.

## Electrical / Low-Power Rules

- `VLT_LCD` drives the active-high OE pin of the `TXU0104RUTR` level translator from MCU 1.8 V signals to display 3.3 V signals.
- `VLT_LCD` high enables the translator. `VLT_LCD` low places translator outputs in high-impedance. The translator must be enabled whenever SPI signals or `LCD_1HZ` / EXTCOMIN must reach the panel.
- `LCD_1HZ` is generated by RTC calibration output and maintains EXTCOMIN/VCOM.
- The display is intended to remain visible in low power.
- Firmware safe default before display ownership is `VLT_LCD` low / translator disabled. Default display-hold policy after initialization is `VLT_LCD` high / translator enabled during hold and updates.
- Experimental policy may duty-cycle `VLT_LCD` only after validation proves image stability, EXTCOMIN correctness, and measurable power savings.

|---|---|

The display contract must define before implementation:

- pixel polarity
- native row order
- native byte order
- command framing
- line address format
- framebuffer packing
- dirty granularity
- DMA alignment and buffer region

## DMA-Safe Buffer Placement

HW5 design requires display DMA/LPDMA source buffers to live in a DMA-safe memory region.

Design target:

| Buffer | Purpose | Expected Size |
|---|---|---|
| final panel framebuffer | composed native `144 x 168` 1bpp panel image | `3024` bytes |
| display transmit payload buffer | Sharp LCD command/row payload stream | about `3362` bytes for full screen, pending final command-format validation |
| precomposed low-power sequence payloads | validated ULP/LPBAM sequence content | profile and SRAM4 budget limited |

SRAM4 is the intended HW5 region for:

- DMA-safe display framebuffer/payload buffers
- precomposed low-power sequence payloads used by LPDMA/LPBAM scenarios
- small retained fast-resume state owned by the runtime/power policy

This is a placement and budgeting requirement, not game-facing API.

Packages and tools do not name SRAM4, buffer addresses, row payload formats, or LPBAM descriptors. They provide validated rendering assets and `precomposed_low_power_sequence` assets through [[Rendering_API_Contract]] and [[Asset_Pipeline_and_Package_Tooling_Contract]].

Exact linker sections, alignment, SRAM4 budget split, and retention behavior must be validated on HW5 and recorded in [[Memory_and_Budgeting_Contract]], [[HW5_DMA_Map]], and [[Brought_Up_Tracker]].

## Update Classes

Display requests must declare an update class:

| Class | Meaning |
|---|---|
| `DISP_UPDATE_STATIC` | one-shot static screen update |
| `DISP_UPDATE_LP_PARTIAL` | low-power partial update |
| `DISP_UPDATE_LP_FULL` | low-power full-frame update |
| `DISP_UPDATE_RT_PARTIAL` | realtime/game partial update |
| `DISP_UPDATE_RT_FULL` | realtime/game full-frame update |
| `DISP_UPDATE_INSTALLER_STATIC` | installer/flashing status screen |
| `DISP_UPDATE_LPBAM_SEQUENCE` | prevalidated low-power autonomous animation |

The previous model of monitoring framebuffer changes and flushing dirty regions remains valid. Update classes give power policy and scheduling clearer intent.

## Display Owner FSM

| State | Meaning |
|---|---|
| `DISP_OFF` | display path inactive; no SPI/LPDMA/LPBAM ownership active |
| `DISP_POWER_ENABLE` | `VLT_LCD` enabled and signal path settling |
| `DISP_INIT` | panel setup, clear, and first-frame path |
| `DISP_STATIC_HOLD` | panel holding image, `LCD_1HZ` maintained, no active flush |
| `DISP_LP_DIRTY` | low-power mode has pending changed rows/regions |
| `DISP_LP_FLUSH` | bounded low-power partial/full flush active |
| `DISP_RT_ACTIVE` | realtime/game mode accepting dirty/full updates |
| `DISP_RT_FLUSH` | realtime flush active |
| `DISP_LPBAM_ARM` | preparing low-power autonomous SPI animation scenario |
| `DISP_LPBAM_RUNNING` | LPBAM linked-list animation running if proven |
| `DISP_LPBAM_EXIT` | stopping LPBAM and reclaiming normal display ownership |
| `DISP_SUSPENDING` | draining/aborting active transfers before mode change |
| `DISP_SUSPENDED` | display owner quiesced, panel hold policy active |
| `DISP_RECOVERING` | bounded recovery after SPI/DMA/LPBAM/display fault |
| `DISP_ERROR` | fatal display fault for normal operation |

## Display Events

| Event | Source | Meaning |
|---|---|---|
| `EV_DISP_ENABLE_REQ` | Platform/display policy | enable display path |
| `EV_DISP_POWER_READY` | `thDisplay` timer/check | translator/display signal path ready |
| `EV_DISP_INIT_OK` | `thDisplay` | first-frame/init path complete |
| `EV_DISP_INVALIDATE` | UI/Engine/runtime | region became dirty |
| `EV_DISP_PRESENT` | UI/Engine/runtime | present request submitted |
| `EV_DISP_FLUSH_DONE` | LPDMA/SPI completion signal | active flush complete |
| `EV_DISP_STATIC_HOLD_REQ` | power/display policy | enter static hold |
| `EV_DISP_RT_ENTER` | runtime/power policy | enter realtime update mode |
| `EV_DISP_LP_ENTER` | runtime/power policy | enter low-power update mode |
| `EV_DISP_LPBAM_REQ` | display/power policy | arm prevalidated LPBAM sequence |
| `EV_DISP_LPBAM_RUNNING` | LPBAM/display signal | autonomous scenario active |
| `EV_DISP_LPBAM_WAKE_REQ` | input/power/runtime event | stop autonomous scenario |
| `EV_QUIESCE` | `thPower` | prepare for sleep/mode transition |
| `EV_RESUME` | `thPower` | resume from sleep/mode transition |
| `EV_DISP_FAULT` | `thDisplay` | SPI/DMA/LPBAM/display fault |
| `EV_RECOVER_OK` | `thDisplay` | recovery completed |

## Display Transition Rules

| Current | Event | Next | Required Action |
|---|---|---|---|
| `DISP_OFF` | `EV_DISP_ENABLE_REQ` | `DISP_POWER_ENABLE` | drive `VLT_LCD` high/enabled, start settle path |
| `DISP_POWER_ENABLE` | `EV_DISP_POWER_READY` | `DISP_INIT` | initialize panel and first framebuffer |
| `DISP_INIT` | `EV_DISP_INIT_OK` | `DISP_STATIC_HOLD` | maintain `LCD_1HZ`, publish ready |
| `DISP_STATIC_HOLD` | `EV_DISP_INVALIDATE` | `DISP_LP_DIRTY` | mark dirty region |
| `DISP_LP_DIRTY` | `EV_DISP_PRESENT` | `DISP_LP_FLUSH` | build bounded partial/full flush |
| `DISP_LP_FLUSH` | `EV_DISP_FLUSH_DONE` | `DISP_STATIC_HOLD` | commit dirty state |
| `DISP_STATIC_HOLD` | `EV_DISP_RT_ENTER` | `DISP_RT_ACTIVE` | accept realtime update cadence |
| `DISP_RT_ACTIVE` | `EV_DISP_PRESENT` | `DISP_RT_FLUSH` | build realtime flush |
| `DISP_RT_FLUSH` | `EV_DISP_FLUSH_DONE` | `DISP_RT_ACTIVE` | commit dirty state |
| `DISP_STATIC_HOLD` | `EV_DISP_LPBAM_REQ` | `DISP_LPBAM_ARM` | validate LPBAM sequence and resources |
| `DISP_LPBAM_ARM` | `EV_DISP_LPBAM_RUNNING` | `DISP_LPBAM_RUNNING` | hand autonomous sequence to LPBAM |
| `DISP_LPBAM_RUNNING` | `EV_DISP_LPBAM_WAKE_REQ` | `DISP_LPBAM_EXIT` | stop/reclaim LPBAM and SPI resources |
| `DISP_LPBAM_EXIT` | `EV_DISP_FLUSH_DONE` | `DISP_STATIC_HOLD` | restore normal owner state |
| any active state | `EV_QUIESCE` | `DISP_SUSPENDING` | drain/abort active transfer safely |
| `DISP_SUSPENDING` | `EV_DISP_FLUSH_DONE` | `DISP_SUSPENDED` | acknowledge quiesce |
| `DISP_SUSPENDED` | `EV_RESUME` | `DISP_STATIC_HOLD` | revalidate signal path and state |
| any state | `EV_DISP_FAULT` | `DISP_ERROR` | publish fatal display fault |
| `DISP_ERROR` | `EV_RECOVER_OK` | `DISP_POWER_ENABLE` | recover through full enable/init path |

Invalid transitions must be rejected and logged.

## Flush Engine FSM

Flush behavior is separate from display mode.

| State | Meaning |
|---|---|
| `FLUSH_IDLE` | no active transfer |
| `FLUSH_BUILD_CMD` | command/line payload being prepared |
| `FLUSH_TRANSFER_ACTIVE` | SPI/LPDMA transfer active |
| `FLUSH_WAIT_COMPLETE` | waiting for completion signal |
| `FLUSH_COMMIT` | dirty map and framebuffer state committed |
| `FLUSH_ABORT` | bounded abort path |
| `FLUSH_ERROR` | transfer/command fault |

Flush rules:

- no overlapping transfers
- partial row/region updates supported
- full-frame fallback supported
- dirty threshold must be deterministic
- all transfers are bounded
- ISR only signals completion/fault
- `thDisplay` owns commit and recovery

## LPBAM Display Scenario FSM

LPBAM display animation is a planned HW5 experiment and must be treated as an authority-gated Platform path.

It is allowed and designed-for, but not required for the normal display renderer to function.

| State | Meaning |
|---|---|
| `LPBAM_DISP_UNAVAILABLE` | no validated LPBAM display scenario available |
| `LPBAM_DISP_PREPARE` | static frame/row sequence and resources being prepared |
| `LPBAM_DISP_VALIDATE_LIST` | linked-list scenario bounds and ownership validated |
| `LPBAM_DISP_ARMED` | scenario ready but not yet running |
| `LPBAM_DISP_RUNNING` | autonomous low-power display animation active |
| `LPBAM_DISP_WAKE_REQUESTED` | input/power/runtime event requests CPU/display owner wake |
| `LPBAM_DISP_STOPPING` | stopping LPBAM and reclaiming SPI/display resources |
| `LPBAM_DISP_FAULT` | LPBAM scenario fault |

LPBAM rules:

- LPBAM may only run prevalidated frame/row sequences.
- LPBAM cannot own arbitrary game rendering.
- LPBAM is intended for low-power idle animations, such as pet idle loops.
- LPBAM sequence payloads must be precomposed and placed in the approved DMA-safe/retained display buffer region before low-power playback.
- CPU wakes when animation class changes, input arrives, storage/power event occurs, or LPBAM faults.
- Normal display owner must reclaim SPI3/LPDMA/LPBAM resources cleanly after exit.
- LPBAM does not replace the normal renderer; it is a low-power idle-animation accelerator.


LPBAM bring-up timing:

- The LPBAM experiment should be attempted after baseline full-frame update, partial update, EXTCOMIN, and static hold are proven.
- It should be attempted before final sleep/wake power integration is closed, so the power architecture can use it if it works.
- It must not block normal display bring-up or normal renderer implementation if it fails.

LPBAM acceptance criteria:

1. prevalidated frame/row sequence renders correctly
2. autonomous transfer runs without CPU intervention for the intended window
3. current draw is measured against normal RTC wake/update strategy
4. input, RTC, PMIC, storage, and fault events can wake/stop the scenario cleanly
5. `thDisplay` reclaims SPI3/LPDMA/LPBAM ownership without stale state
6. image correctness and EXTCOMIN behavior remain valid
7. failure routes to normal display owner recovery or disables LPBAM without breaking normal rendering

## Dirty-Region Policy

Dirty update policy must define:

- dirty granularity, likely rows or row-ranges first
- threshold for full-frame fallback
- maximum low-power update budget per cycle
- orientation transform cost and ownership
- static hold behavior after a flush

Policy must be deterministic and bounded.

## Low-Power Behavior

- The display is an always-visible status surface.
- `LCD_1HZ` / EXTCOMIN must be maintained while the panel is holding an image.
- `VLT_LCD` remains high/enabled during hold/update by default.
- `VLT_LCD` duty-cycling is experimental and requires evidence.
- Static installer/flashing screens should be presented once and then held with minimal activity.
- LPBAM scenarios may run only after validation and must be recoverable.

## Failure Policy

Display failure is fatal for normal operation.

On display failure:

- normal shell/game operation must not continue as if display is usable
- fault/safe path must be entered
- diagnostics should be exposed through debug/USB if available
- recovery may be attempted through full display re-enable/init path

## Validation Cases

1. first-frame bring-up correctness on LS013B7DH05
2. logical landscape `168 x 144` maps correctly to native portrait `144 x 168`
3. pixel polarity, row order, line address, and byte ordering are verified
4. partial row/region update correctness
5. full-frame fallback correctness
6. no overlapping SPI/LPDMA transfers
7. `LCD_1HZ` / EXTCOMIN remains correct during static hold
8. `VLT_LCD` high/enabled hold policy preserves image
9. optional `VLT_LCD` duty-cycle experiment records image stability and power evidence
10. suspend/resume leaves no stale transfer state
11. installer/flashing static screen can hold with minimal activity
12. LPBAM idle animation experiment is attempted after baseline display validation and before sleep/wake integration closure
13. LPBAM exit cleanly returns ownership to `thDisplay`
14. display fault routes to fatal normal-operation fault path
